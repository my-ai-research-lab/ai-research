# link-skill-dojo 踩坑经验汇总

> 从记忆系统迁移至此。
> 最后更新：2026-03-22

## 1. 一步到位执行修复原则（只诊断不修复是反模式）

**问题**：skill-dojo 诊断出问题后，只输出"建议"而不执行修复。  
**根因**：诊断和修复是分开的，诊断完就停了，没有闭环。  
**规则**：
- 技能管理必须一步到位，发现问题立即修复，不能只输出清单
- skill-dojo 报告必须用 HTTP URL（不能用 file:// 协议），防止用户无法打开

**检查点**：
- [ ] 诊断结果是否都有对应的修复动作？
- [ ] 报告链接是否是 HTTP URL（非 file://）？

## 2. 方式B必须启动独立Agent会话（v3：含每日修炼手动续跑场景）

**问题（三代踩坑）**：

| 版本 | 错误方式 | 危害 |
|:-----|:---------|:-----|
| v1 | 在当前会话中手动测试技能，输出模拟结果 | 非真实独立会话，结果无效 |
| v2 | 用 agent-session-controller 分 Cat，但编排会话串行触发 11 个 Cat | 编排会话自身 context 耗尽，Cat 5-11 全未执行 |
| v3 | 续跑时在当前会话直接 `use_skill` 执行 Step 4 | 复用旧 evidence.json 通过门控，实际未执行 |

**强制规则（v3）**：
```
凡是 Skill 文档标注了"独立会话"的步骤，无论何种触发方式（自动/手动/续跑），
都必须启动独立 Agent Session，禁止在当前会话直接执行。
```

**批量阈值**：
- 技能总数 ≤ 10：单会话可完成
- 11-30：按 Cat 分组，每组独立会话
- > 30：每 Cat 一个独立会话，用脚本（multi-session-runner.sh）编排

**续跑守卫（v3 新增）**：
- 续跑时必须先读技能文档确认执行方式，不能凭记忆直接执行
- 检查 evidence.json 的 generated_at 是否在当前任务启动后（防复用旧文件）

**根因**：Skill 文档的执行方式约束比记忆更严格，续跑会惯性跳过。
