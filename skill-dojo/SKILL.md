# 技能修炼场（Skill Dojo）

> **定位**：技能生命周期的第四维度 — 持续精进。
>
> ```
> link-find-skills (发现) ──→ skill-creator (创建) ──→ skill-evaluator (评估) ──→ skill-dojo (修炼)
>      ↑                                                                         │
>      └────────────────── 社区对标触发重新发现 ──────────────────┘
> ```
> 
> skill-evaluator 回答“这个技能质量怎么样”，skill-dojo 回答“这个技能怎么才能更好”。

---

## 核心理念

```
技能不是写完就好了。
写完的技能 = 入门级；
经过修炼的技能 = 大师级。

修炼 = 不断发现弱点 + 不断改进 + 不断验证
```

**类比**：技能修炼场就像代码的 CI/CD 流水线 —— 代码写完要跑测试、要持续集成、要监控线上表现；技能也一样，要跑场景测试、要持续改进、要监控执行表现。

## 六大能力

### 能力 1: 场景库管理

每个技能可以维护一组测试场景，用于持续评测和回归检测。

**场景来源**：

| 来源 | 说明 | 自动/手动 |
|:-----|:-----|:----------|
| 创建时定义 | skill-creator 创建技能时的 test cases | 手动 |
| 历史对话提取 | 从每日修炼的对话复盘中提取 | 半自动 |
| 踩坑记忆生成 | 从 common_pitfalls_experience 中生成边界用例 | 半自动 |
| 用户手动添加 | 用户发现新的测试场景 | 手动 |

**场景格式**：

```json
{
  "scenario_id": "sc-001",
  "skill_name": "ai-insight",
  "display_name": "ai-insight（AI日报）",
  "description": "生成包含5条新闻的AI日报",
  "prompt": "跑一下今天的AI日报",
  "expected_behaviors": [
    "至少收集5条AI新闻",
    "每条新闻有来源链接",
    "生成HTML页面并部署"
  ],
  "assertions": [
    {"type": "contains", "name": "包含链接", "value": "http"},
    {"type": "not_contains", "name": "无报错", "value": "Traceback"}
  ],
  "artifact_extractors": [
    {"name": "部署链接", "type": "url", "pattern": "https?://[^\\s]+\\.html"}
  ],
  "source": "historical_conversation",
  "priority": "P1",
  "timeout": 300
}
```

> ⚠️ **场景文件编写规范**（从踩坑中提炼）：
> 1. **禁止在 JSON 字符串值中使用中文全角引号 `"…"`** — 会导致 JSON 解析失败。用书名号 `《》` 或直角引号 `「」` 代替
> 2. **assertions 字段必须用 `name`（非 `target`）** — runner 按 `name` 标识断言，用 `target` 会被忽略
> 3. **必填字段**：`scenario_id`、`skill_name`、`display_name`、`prompt`、`priority`、`timeout`
> 4. **timeout 建议值**：纯文本/翻译类 120s；生图/截图/调研类 300s；多工具串联类 600s
> 5. **新场景文件创建后先跑 `--dry-run` 验证**，确认 JSON 合法且场景数量正确

**存储位置**：场景库不在技能目录内（避免污染技能本体），统一存储在：
```
~/Documents/Codeflicker/[WORKSPACE]/knowledge/self-evolution/skill-dojo/
├── scenarios/
│   ├── ai-insight/
│   │   ├── sc-001.json
│   │   └── sc-002.json
│   ├── web-dev-workflow/
│   │   └── sc-001.json
│   └── ...
├── reports/
│   ├── weekly-2026-03-14.json
│   └── ...
└── baselines/
    ├── ai-insight.json
    └── ...
```

### 能力 2: 自动化评测

**流程**：

```
选择技能 → 加载场景库 → 逐场景评测 → 打分 → 生成报告
```

**评测维度**：

| 维度 | 权重 | 说明 |
|:-----|:-----|:-----|
| 功能正确性 | 40% | assertions 通过率 |
| 执行效率 | 20% | 完成时间、token消耗 |
| 用户满意度 | 20% | 历史返工次数、用户纠正次数 |
| 边界覆盖 | 20% | 边界场景的处理质量 |

**评测方式（三种）**：

**方式A: 轻量评测（推荐日常使用）**
- 从场景库中选取场景
- 人工走读：回顾最近该技能的执行日志
- 对照 expected_behaviors 逐条评分（1-5分）
- 输出评分卡

> **方式A/B 混用补全策略**（2026-03-19 实战验证）：
> 当方式B自动化评测因竞态被截断时，用方式A补全结论：
> - 已有 transcript（无论完整度）→ 读取实际输出验证部分断言
> - 无 transcript → 查 SKILL.md 文档规范 + 历史使用记录验证
> - 最终报告标注评测来源（方式A/B/混合），综合通过率按方式A结论计算

**方式B: 深度评测（周度/月度使用）**

> ⚠️ **方式B 强制规则（P0 级，2026-03-20 踩坑沉淀）**
>
> 当用户要求"方式B"/"深度评测"/"批量评测"时，**必须**：
> 1. 使用 `skill-test-runner.py` 脚本执行（推荐）
> 2. 或使用 `agent-session-controller` 为每个技能/分类创建独立会话
> 3. **禁止**在当前会话逐个手动测试
>
> **为什么必须用独立会话**：
> - 避免上下文污染：每个技能评测独立，互不干扰
> - 符合设计：方式B 设计就是 subagent 执行
> - 可追溯：每个会话有独立 transcript，可复盘
> - 真实性：独立会话才是真正的"技能被触发后的表现"
>
> **检查点（启动前必须确认）**：
> - [ ] 是否调用了 `skill-test-runner.py`？
> - [ ] 或是否调用了 `session-create.sh` 创建独立会话？
> - [ ] 当前会话是"编排者"还是"执行者"？（方式B中当前会话应是编排者）
>
> **正确的会话角色分工**：
> ```
> 当前会话(编排者)
>     ├── 创建 Cat1 会话 → 评测6个技能 → 返回结果
>     ├── 创建 Cat2 会话 → 评测10个技能 → 返回结果
>     └── 汇总所有会话结果 → 生成最终报告
> ```
>
> **错误模式（严禁）**：
> ```
> 当前会话(伪装成编排者)
>     └── 在本会话逐个手动测试77个技能 → 输出模拟结果
>     （这不是方式B，这是偷换概念！）
> ```

已落地为完整的自动化执行引擎（v1.5 新增），脚本位于 `~/.codeflicker/skills/link-skill-dojo/scripts/`：

| 脚本 | 职责 |
|:-----|:-----|
| `skill-test-runner.py` | 主控引擎：从 JSON 场景文件批量启动 agent 会话、收集结果、生成 benchmark.json |
| `graders/assertion_grader.py` | 断言评分器：解析 grading.json，判断每条 assertion 通过/失败 |
| `graders/artifact_collector.py` | 交付物收集器：从对话记录中提取文件路径、URL、图片等 |
| `report_generator.py` | HTML 报告生成器：把 benchmark.json 渲染成带样式的评测报告表格 |
| `transcript_renderer.py` | 对话复盘渲染器：把对话记录 JSON 渲染成 HTML 复盘页（带折叠思考、工具调用高亮）|

**CLI 参数**（`skill-test-runner.py`）：

```bash
# 运行默认场景库（scenarios/baseline-v1.json）
python3 skill-test-runner.py --all

# 指定自定义场景文件（必须用 --file/-f 参数）
python3 skill-test-runner.py --file ../scenarios/execution-layer-v1.json --all

# dry-run 验证场景文件是否可正确加载
python3 skill-test-runner.py --file ../scenarios/my-scenarios.json --all --dry-run

# 按优先级或技能名过滤
python3 skill-test-runner.py --file ../scenarios/my-scenarios.json --priority P1
python3 skill-test-runner.py --file ../scenarios/my-scenarios.json --skills baoyu-translate,pptx
```

> ⚠️ 注意：`--scenarios` 不是有效参数（容易混淆），正确参数是 `--file/-f`

**执行流程**：
```
scenarios/*.json
    ↓ skill-test-runner.py (为每个 scenario 创建独立 agent session)
    ↓ 每个 session 执行任务 → messages 存入 transcripts/{scenario_id}.json
    ↓ 内置 assertion_grader + artifact_collector 分析输出文本
    ↓ benchmark.json（结构化结果汇总）
    ↓ report_generator → benchmark-{date}.html（可交互评测报告）
    ↓ transcript_renderer → {scenario}.html（每个场景对话复盘页）
```

> ⚠️ **竞态陷阱（P1）**：task-wait.sh 在 Agent 执行工具调用的间隙可能误判 `isRunning=false` 为完成。
> 症状：场景在 10-30s 内「完成」，但 transcript 只有 2-3 条初始消息。
> 根因：工具调用执行期间存在短暂的 isRunning=false 窗口（详见记忆：「Agent定时任务假完成陷阱 v3」）。
> 临时规避：对 isRunning=false 连续3次才认为完成；grace period 建议 >=30s。
> 若遭遇截断：用方式A（走读补全）+已有 transcript 补全评估结论。
> ⚠️ **方式B必须独立会话（含续跑场景）**：详见 `references/lessons-learned.md` §2

**报告系统**（v5.2 — 模板化重构 + 链接修复）：

> 报告样式已独立为 `templates/` 子模块，详见 `templates/README.md`。

**两种报告类型**：

| 报告 | 生成器 | 模板 | 用途 |
|:-----|:-------|:-----|:-----|
| 评测报告 | `HTMLReportGenerator` | `benchmark-report.html` | 方式B自动化评测结果 |
| 改进报告 | `ImprovementReportGenerator` | `improvement-report.html` | 方式A轻量评测+修复记录 |

**报告结构**：
- 统计卡片（通过率/通过/失败/错误）
- 技能芯片（颜色区分状态，点击跳转）
- 结果表格（技能/状态/评测项/交付物/耗时）

> ⚠️ **HTTP 服务器规范**：
> - 根目录设为 `link-skill-dojo/`（不是 `reports/`）
> - 启动：`python3 -m http.server 8765 --directory ~/.codeflicker/skills/link-skill-dojo/`

> ⚠️ **链接协议规范**（v5.2 踩坑沉淀）：
> - **禁止 `file://` 协议**：浏览器安全策略阻止 HTTP 页面中的 `file://` 链接跳转
> - **必须使用相对路径**：交付物和会话记录链接用 `../transcripts/xxx` 格式
> - **report_generator.py 已修复**：v2.0 版本自动生成相对路径，无需手动处理
> - 验证方法：grep 报告 HTML 中是否存在 `file://`，若有则链接不可点击

**benchmark.json 数据规范**：
- artifacts 字段必须用 `value` 而非 `path`
- 所有链接必须是 HTTP URL（禁止 file:// 协议）
- transcript_path 用绝对路径（脚本会自动转换为相对路径）

**方式C: 对照评测（v1.4新增，吸收自万擎skill-creator）**
> 核心思想：不只是评估技能好不好，而是对比"有技能"vs"无技能"的差异。

```
subagent 并行执行
├── with-skill: 加载技能执行任务
└── baseline: 不加载技能执行同一任务
     ↓
结果存入 iteration-N/eval-ID/ 目录
     ↓
聚合到 benchmark.json
```

**benchmark.json 量化指标**（v1.4新增）：

| 指标 | 说明 | 计算方式 |
|:-----|:-----|:---------|
| pass_rate | 断言通过率 | passed / total assertions |
| time_ms | 执行时间 | duration_ms (mean ± stddev) |
| tokens | Token消耗 | total_tokens (mean ± stddev) |
| delta | 与基线差异 | with_skill - baseline |

**grading.json 断言格式**：
```json
{
  "expectations": [
    {"text": "断言描述", "passed": true, "evidence": "通过证据"}
  ]
}
```

**对照评测适用场景**：
- 评估新技能是否真正提供价值
- 对比技能改进前后的效果
- 量化技能的 ROI（时间/token 投入 vs 质量提升）

### 能力 3: 退化检测

**核心机制**：对比历史基线，发现技能表现下降。

```
每次评测结束
    ↓
对比 baselines/{skill-name}.json 中的历史评分
    ↓
当前评分 < 历史基线 × 0.9 → 触发退化预警
    ↓
退化预警 → 自动标记为"需要改进" → 触发改进闭环
```

**基线更新规则**：
- 新技能首次评测 → 直接设为基线
- 评分提升 → 更新基线
- 评分下降 → 不更新基线，触发告警
- 手动重置 → 用户确认后可重置基线

### 能力 4: 弱点发现

**被动发现（从日常执行中）**：

| 信号 | 来源 | 含义 |
|:-----|:-----|:-----|
| 返工 >= 2次 | 每日修炼对话复盘 | 技能指引不够清晰 |
| 用户纠正 | 对话中用户的修正反馈 | 技能规则有遗漏 |
| 执行失败 | 脚本报错/流程中断 | 技能对边界处理不足 |
| 跳过技能 | 本该触发但没触发 | description 触发词不够准确 |

**主动发现（通过评测）**：

| 信号 | 来源 | 含义 |
|:-----|:-----|:-----|
| assertion 失败 | 场景评测 | 功能缺陷 |
| 评分大幅波动 | 多次评测对比 | 稳定性问题 |
| 某类场景持续低分 | 场景分类分析 | 特定领域弱项 |

**弱点输出格式**：
```
🔍 弱点报告: {skill-name}
├── 弱点类型: {功能缺陷/边界遗漏/指引不清/触发不准}
├── 证据来源: {对话ID/评测报告/用户反馈}
├── 具体描述: {一句话说清楚问题}
├── 影响范围: {高/中/低}
└── 建议改进: {具体的修改建议}
```

### 能力 5: 改进闭环

**两种模式**：

| 模式 | 触发方式 | 修复行为 | 适用场景 |
|:-----|:---------|:---------|:---------|
| 交互模式 | 手动触发（用户说"技能修炼"） | 生成建议 → 用户确认 → 执行 | 重大改动、不确定的修改 |
| 自动模式 | 定时任务（周检/月检/每日修炼） | 直接执行修复 | P0/P1级别的明确问题 |

**自动修复条件（v1.3 新增）**：

满足以下**全部**条件时，定时任务中可自动修复，无需用户确认：

1. **问题类型明确**：属于以下可自动修复类型
   - 文档过时（版本号/日期/数量不一致）
   - 触发词缺失（常见触发词未覆盖）
   - 格式问题（缩进/换行/标点）
   - 冗余内容（重复段落/废弃引用）

2. **影响范围可控**：修改不超过 20 行

3. **有明确证据**：来自评测/日志的具体问题，非推测

4. **可回滚**：技能在 git 版本控制下

**禁止自动修复的情况**：
- 逻辑变更（流程/规则/判断条件）
- 新增功能（新的 section/能力）
- 删除内容（除非是明确的冗余）
- 涉及其他技能的联动修改
- 平台预装技能（只读）

**改进流程**：

```
发现弱点（内部评测 或 社区对标）
    ↓
分类问题
    ├── 可自动修复类 → 检查自动修复条件
    │       ├── 满足全部条件 + 定时任务触发 → 直接执行修复
    │       └── 不满足或手动触发 → 生成建议等待确认
    └── 不可自动修复类 → 生成建议等待用户确认
    ↓
执行改进（调用对应工具）
    ├── 文档修改 → replace_in_file
    ├── 新增文件 → write_to_file
    ├── 触发词优化 → 更新 SKILL.md frontmatter
    └── 版本升级 → 触发吸星大法模式B（定向版本升级）
    ↓
回归测试（重跑改进前失败的场景）
    ↓
确认改进有效 → 更新基线
```

**自动修复执行格式**：

```
🔧 自动修复执行 | 技能: {skill-name}
├── 问题: {一句话描述}
├── 类型: {文档过时/触发词缺失/格式问题/冗余内容}
├── 修改: {具体改了什么}
├── 行数: {N}行
└── 状态: ✅ 已修复 / ❌ 修复失败（原因）
```

### 能力 6: 社区对标（Community Benchmarking）

> skill-dojo 从 v1.2 起不再只做"内视型"修炼。
> 能力 1-5 是闭门练功，能力 6 是出门看看别人怎么练的。

**核心逻辑**：

```
定期巡检（调用 link-find-skills 主动巡检模式）
    ├── skills.sh / ClawHub → 已有技能是否有新版本？同类技能做得怎么样？
    ├── ClawBook #til / #builds → 有没有人分享了和现有技能相关的更优实践？
    └── ClawBook #evolution → 其他 Claw 的进化方法有什么可借鉴的？
    ↓
对标分析（不是发现了就吸收，是和自己对比）
    ├── 外部版本 vs 自己版本 → 哪些 diff 有价值？
    ├── 社区最佳实践 vs 自己实践 → 差距在哪？
    └── 是否有自己完全没想到的方向？
    ↓
输出「社区对标报告」→ 汇入改进闭环（能力 5）
```

**与吸星大法的分工**：

| | 社区对标（skill-dojo 能力6） | 吸星大法 |
|:--|:--|:--|
| 目标 | 已有技能的持续升级 | 全新技能的发现与吸收 |
| 搜索范围 | 同名/同类技能 + 社区讨论 | 全量元技能关键词巡检 |
| 触发方式 | 周度自动（随修炼节奏） | 月度主动 或 用户触发 |
| 产出 | 升级候选清单 → 改进闭环 | 精华卡片 → 融入四棵树 |
| 调用关系 | 调用 link-find-skills 主动巡检模式 | 调用 link-find-skills 搜索模式 |

**社区对标的三个信号源**：

| 信号源 | 工具 | 关注点 |
|:-------|:-----|:-------|
| 技能市场版本 | link-find-skills 主动巡检 | 已安装技能在 skills.sh / ClawHub 是否有新版本或更优替代 |
| 社区经验帖 | ks-clawbook 浏览 #til / #builds | 其他 Claw 分享的与已有技能相关的实践经验 |
| 社区进化讨论 | ks-clawbook 浏览 #evolution | 其他 Claw 的修炼方法、框架升级 |

**社区帖子筛选标准**：

```
值得关注 = upvote >= 3 且 (板块 in [#til, #builds, #evolution])
           且 (内容与已有技能相关 或 涉及自进化方法论)

不关注   = 过于细碎的小技巧（如单条命令）
           或 与已有体系完全无关的应用层话题
```

**对标报告格式**：

```
## 社区对标报告 [{日期}]

### 技能市场发现
| 已有技能 | 外部版本 | 对比结果 | 建议 |
|:---------|:---------|:---------|:-----|
| {name} | v{x.y} → v{a.b} | {关键diff} | 升级/观望/忽略 |

### 社区经验发现
| 帖子标题 | 作者 | 赞数 | 关联技能 | 可借鉴点 |
|:---------|:-----|:-----|:---------|:---------|
| {title} | {author} | {n} | {skill} | {一句话} |

### 行动建议
- P0: {紧急升级项}
- P1: {本周改进项}
- P2: {月度关注项}
```

## 执行规则

### Rule 1: 修炼节奏

| 频率 | 内容 | 触发方式 | 修复模式 |
|:-----|:-----|:---------|:---------|
| 日常 | 被动收集弱点信号 | 每日修炼自动标记 | 自动（满足条件时） |
| 周度 | **本周有调用（≥1次）的所有技能**轻量评测 + 社区对标（本周0次调用的技能跳过）| 小无相功周检调用 | 自动（满足条件时） |
| 月度 | 全量技能深度评测 + 全量社区对标 | 小无相功月检调用 | 自动（满足条件时） |
| 即时 | 用户说"技能修炼" | 手动触发 | 交互（需确认） |

### Rule 2: 修炼优先级

当有多个技能需要改进时：

```
优先级决策树：
  使用频率高 + 退化明显 → P0（立即改进）
  使用频率高 + 轻微问题 → P1（本周改进）
  使用频率低 + 退化明显 → P1（本周改进）
  使用频率低 + 轻微问题 → P2（月度改进）
```

### Rule 3: 不修炼什么

- 外部预装技能（如 vercel-react-best-practices）→ 只评测不修改（但社区对标可发现其更新）
- 一次性使用的技能 → 不建场景库
- 刚创建的技能（< 1周） → 先积累执行数据再评测

## 与其他技能的关系

```
技能生命周期全链条（v1.2: 已成环）：

  link-find-skills ──→ skill-creator ──→ skill-evaluator ──→ skill-dojo
  (发现)          (创建)            (质量评估)          (持续修炼)
     ↑                                                    │
     │                                                    │
     └──────────── 社区对标触发重新发现 ──────────────────┘
                   （能力6 调用 link-find-skills 主动巡检模式）
```

**与 link-find-skills 的关系**：
- skill-dojo 能力6（社区对标）调用 link-find-skills 主动巡检模式，检查已有技能的外部更新
- link-find-skills 返回升级候选清单 → 汇入 skill-dojo 改进闭环
- 这条反馈路径让技能生命周期从线性链变为真正的闭环

**与吸星大法的分工**：
- **吸星大法**：发现和吸收**全新的**外部元技能（我没有的）
- **skill-dojo 社区对标**：检查**已有技能**是否有更好的外部实践（我有但可能落后了）
- 边界情况：如果社区对标发现的不是「升级已有技能」而是「需要全新技能」→ 转交吸星大法

**与 ks-clawbook 的关系**：
- skill-dojo 能力6 调用 ks-clawbook 浏览 #til / #builds / #evolution 板块
- 从社区帖子中发现可借鉴的实践 → 筛选 → 汇入改进闭环
- ks-clawbook 负责社区互动（发帖/评论/投票），skill-dojo 负责从社区**学习**

**与每日修炼的关系**：
- 每日修炼的对话复盘 → 自动产生弱点信号给 skill-dojo
- skill-dojo 的周度评测 → 结果汇入每日修炼的技能健康度指标

**与小无相功的关系**：
- 小无相功周检 → 调用 skill-dojo 跑轻量评测 + 社区对标
- 小无相功月检 → 调用 skill-dojo 跑全量评测 + 全量社区对标

## 典型对话流程

### 场景 1: 手动触发修炼

**用户**: "技能修炼"

**Agent**:
1. 检查最近7天的执行日志
2. 列出有弱点信号的技能
3. 按优先级排序
4. 询问用户要修炼哪个
5. 加载场景库 → 评测 → 出报告 → 提改进建议

### 场景 2: 周度评测 + 社区对标

**小无相功周检触发**:
1. 从 skill-usage-stats.json 或当周执行日志中，筛选出**本周调用次数 ≥1 次的所有技能**（调用0次的跳过，不浪费修炼预算）
2. 对上述技能逐个进行轻量评测（方式A）
3. 对比各技能历史基线，识别退化信号
4. **社区对标**：调用 link-find-skills 主动巡检 + ks-clawbook 浏览相关板块
5. 输出周度技能健康报告（含社区对标报告）
6. 标记退化技能 + 升级候选技能

> **选取依据**：周度修炼频率低（每周一次），应尽量覆盖所有本周活跃技能，而非只盯 Top N。
> 本周未被调用的技能说明暂无使用场景，不值得消耗修炼预算，下次有调用再纳入。

### 场景 3: 改进闭环（交互模式）

**Agent 发现 ai-insight 趋势分析得分下降**:
1. 报告弱点：趋势分析场景 assertion 失败
2. 分析原因：SKILL.md 中趋势分析规则可能过时
3. 生成建议：更新 ai-insight 技能的趋势分析规则
4. 用户确认后执行
